1.1 Fachliche Argumentation über Erflogsprinzipien des WWW

Mit welchen fachlichen Argumenten wurde das WWW-Proposal von TBL abgelehnt?

Das WWW-Proposal wurde abgelehnt, weil TBL sich unter Hypertext ertwas anderes vorgestellt hatte. Und zwar etwas das sauber ist, wo es keine broken links gibt und wo die Hyperlinks immer stimmen. Sie waren davon überzeugt dass die Hypertext-Community zu dem Zeitpunkt schon weiter vorangeschritten war und dass das Konzept von Tim Berners Lee ein Rückschritt gewesen wäre.

Was sind die fachlichen Argumente, warum das WWW dennoch ein Erfolg wurde?

Der Grund für den Erfolg des WWW ist, dass es auf drei orthogonalen Technologien basiert (HTTP, HTML, URI), die unabhängig von einander weiterentwickelt werden können.

Was wäre der Preis für die garantierte Verhinderung von “broken links”?

Der Preis für die Verhinderung von broken links, wäre eien zentrale Kontrolle und damit die Geschlosseheit des WWW. Das Konzept, dass das WWW für jeden zugänglich ist wäre somit verworfen.

1.2. HTTP

Sie bekommen im Browser den HTTP Status Code 200. Was bedeutet das?

Der HTTP Status Code 200 bedeutet, dass der Zugriff auf eine Ressoure erfolgreich war.

Sie bekommen im Browser den HTTP Status Code 301. Was hat das zu bedeuten?

Der HTTP Status Code 301 gibt an, dass eine Ressource permanent verschoben wird und eine permanente Weiterleitung erfolgt.

Sie bekommen im Browser den HTTP Status Code 400. Was hat das zu bedeuten? Was können Sie dagegen tun?

Der HTTP Status Code 400 bedeutet, dass die Anfrage fehlerhaft war. Um das zu verhindern, kann man die Anfrage und insbesondere die übermittelte URl nochmal überprüfen.

Sie bekommen im Browser den HTTP Status Code 403. Was hat das zu bedeuten? Was können Sie dagegen tun?

Der HTTP Status Code 403 bedeutet, dass der Zugriff auf eine bestimmte URL zugriffsbeschränkt, und somit verboten ist. Eine Möglichkeit wäre die Firewall-Einstellungen zu überprüfen.

In einer Webanwendung benötigen Sie eine OPTIONS-Anfrage, die die Optionen des Servers vor dem eigentlichen Zugriff erfragen soll. Aus Performanzgründen soll die Abfrage jedoch cacheable sein. Wie könnten Sie dafür eine Lösung angehen?

Eine Möglichkeit wäre, die Optionen des Servers ebenfalls mit einem HTTP GET Request zu erfragen, womit diese auch cacheable wären.
